# DIC-SoC 项目最终报告

> 版本：v9 | 日期：2025-07

## 第1章 项目概述与设计目标

本项目设计并实现一个基于 RISC 风格 Load/Store 接口的片上系统（SoC），集成多级缓存层次、AXI4互联、DMA引擎、MMIO寄存器堆及轻量级中断控制器。

**设计目标：**
- 支持 CPU 高速访存，通过 L1/L2 两级缓存降低平均访存延迟
- 提供 DMA 卸载机制，实现零CPU干预的内存块搬移
- 通过 MMIO 寄存器实时采集 L1I/L1D/L2 性能计数
- AXI4 互联支持多主多从、乱序、Outstanding 事务
- 全RTL可综合，适配 DC（500MHz）与 Vivado（100MHz Artix-7）

**地址解码逻辑：**

## 第4章 CPU接口与访存协议

CPU 采用 RISC 风格 Load/Store 接口，与 L1 ICache/DCache 直接相连。

**ICache接口信号：**
```
输入：
  cpu_iaddr  [31:0]  — 取指地址
  cpu_ireq          — 取指请求
输出：
  cpu_idata  [31:0]  — 指令数据
  cpu_ihit          — 命中标志（1=有效数据）
```

**DCache接口信号：**
```
输入：
  cpu_daddr  [31:0]  — 数据地址
  cpu_dwe           — 写使能
  cpu_dwdata [31:0]  — 写数据
  cpu_dwstrb [3:0]   — 字节使能
  cpu_dreq          — 数据请求
输出：
  cpu_drdata [31:0]  — 读数据
  cpu_dhit          — 命中/完成标志
```

**时序示意（读命中）：**
```
clk:    _/‾\_/‾\_/‾\_/‾\
ireq:   _/‾‾‾‾‾‾‾\_____
iaddr:  ──[ADDR]────────
ihit:   ____/‾‾‾\_______   ← 命中时T+1返回
idata:  ────[DATA]──────
```

## 第5章 L1 Cache 设计

### 5.1 L1 ICache 参数

| 参数 | 值 |
|------|----|
| 容量 | 2 KB |
| 组相联度 | 4路 |
| 缓存行大小 | 8 word（32字节） |
| 组数 | 16 组 |
| Tag宽度 | 32-4-4=24 bit |
| 替换策略 | PLRU（伪LRU） |
| 写策略 | 只读（Read-Only） |

### 5.2 L1 DCache 参数

| 参数 | 值 |
|------|----|
| 容量 | 2 KB |
| 组相联度 | 2路 |
| 缓存行大小 | 8 word（32字节） |
| 组数 | 32 组 |
| Tag宽度 | 32-5-4=23 bit |
| 替换策略 | LRU |
| 写策略 | 写回+写分配 |
| NC bypass | addr[31:28]!=0 直接走AXI |

### 5.3 Tag/Data 阵列结构

```
DCache (2路, 32组, 8字/行)
┌─────────┬──────────────┬───────────────────────────────┐
│  valid  │   tag[22:0]  │     data[8×32-1:0]            │
├─────────┼──────────────┼───────────────────────────────┤  Way 0
│  valid  │   tag[22:0]  │     data[8×32-1:0]            │
├─────────┼──────────────┼───────────────────────────────┤  Way 1
  ← 按组索引 [9:5] 寻址，按Tag [31:10] 比较，按偏移 [4:2] 选字 →
```

### 5.4 命中判定

```verilog
wire [22:0] req_tag  = cpu_addr[31:9];
wire [4:0]  req_set  = cpu_addr[8:4];
wire [2:0]  req_woff = cpu_addr[3:2] >> (注: offset内字选择);

wire hit_w0 = valid[req_set][0] && (tag_arr[req_set][0] == req_tag);
wire hit_w1 = valid[req_set][1] && (tag_arr[req_set][1] == req_tag);
wire hit    = hit_w0 | hit_w1;
```

### 5.5 DCache Miss 状态机

```
         ┌─────────────────────────────────────────┐
         │                                         │
         ▼                                         │
       IDLE ──[NC地址]──▶ NC_BYPASS ──▶ NC_WAIT ──┘
         │
         │[miss, dirty=0]
         ▼
      RD_REQ ──▶ RD_WAIT ──▶ FILL ──▶ IDLE
         │
         │[miss, dirty=1]
         ▼
      WB_REQ ──▶ WB_WAIT ──▶ RD_REQ（回写完再填充）
```

### 5.6 写策略

- **写命中**：直接更新Data阵列，置dirty位，不写穿到L2
- **写缺失**：写分配（Write-Allocate）——先从L2填充缓存行，再合并写入
- **替换时**：若dirty=1，先将脏行写回L2（Write-Back），再填充新行

## 第6章 L2 Cache 设计

### 6.1 参数

| 参数 | 值 |
|------|----|
| 容量 | 8 KB |
| 组相联度 | 4路 |
| 缓存行大小 | 16 word（64字节） |
| 组数 | 32 组 |
| Tag宽度 | 32-5-6=21 bit |
| 替换策略 | PLRU |
| 写策略 | 写回 |

### 6.2 Miss 状态机（含 FILL_DONE 修复）

```
        IDLE
          │
          ├─[hit]──────────────────────────────▶ IDLE（返回数据）
          │
          ├─[miss, dirty=0]
          ▼
       RD_REQ ──▶ RD_WAIT ──▶ RD_R（写data_arr）
                                    │
                                    ▼
                              FILL_DONE（读data_arr返回）──▶ IDLE
          │
          ├─[miss, dirty=1]
          ▼
       WB_REQ ──▶ WB_WAIT ──▶ RD_REQ（续上）
          │
          ├─[NC地址]
          ▼
       NC_RD/WR_REQ ──▶ NC_WAIT ──▶ IDLE
```

> **FILL_DONE状态说明**：修复了原RD_R中NBA写入与同周期读出的竞争。写入data_arr在RD_R完成，FILL_DONE下一拍读出并返回上游，消除X态。

## 第7章 L2仲裁模块

### 7.1 独立模块设计原因

L2 Cache 仅有一个上行端口，L1I 与 L1D 均可发起缺失请求，需仲裁。将仲裁逻辑独立为 `l2_arbiter.v` 的原因：
1. **关注点分离**：L2 Cache 专注缓存逻辑，不混入仲裁状态
2. **可替换性**：仲裁策略（round-robin/priority）可独立替换，不影响L2内部
3. **可验证性**：仲裁公平性可单独验证

### 7.2 Round-Robin 算法

```verilog
// last_grant: 上一次授权方（0=L1I, 1=L1D）
always @(posedge clk or negedge rst_n) begin
    if (!rst_n) last_grant <= 1'b0;
    else if (grant_done) last_grant <= cur_grant;
end

// 轮询优先级：上次授权方优先级降低
assign cur_grant = (req_d && (!req_i || last_grant==0)) ? 1'b1 : // L1D
                   (req_i)                               ? 1'b0 : // L1I
                                                           1'bx;
```

### 7.3 Grant 锁存

一旦发出 grant，持续到当前事务结束（L2返回ack），防止中途切换破坏事务完整性：
```verilog
assign grant_locked = grant_active ? cur_grant_reg : cur_grant;
```

### 7.4 公平性证明

在持续双发请求场景下，轮询保证 L1I 与 L1D 各获得 ≤1次连续授权，最大等待时间 ≤ 1个L2访问周期（含缺失填充）。

## 第8章 AXI4/AXI4-Lite 互联

### 8.1 通道与握手

AXI4 五通道：
```
AR（读地址）: ARVALID/ARREADY/ARADDR/ARLEN/ARSIZE/ARBURST/ARID
R （读数据）: RVALID/RREADY/RDATA/RRESP/RID/RLAST
AW（写地址）: AWVALID/AWREADY/AWADDR/...
W （写数据）: WVALID/WREADY/WDATA/WSTRB/WLAST
B （写响应）: BVALID/BREADY/BRESP/BID
```

握手规则：valid/ready 双向握手，valid 不得依赖 ready（AXI4 规范）。

### 8.2 地址解码

```
if      (addr[31:28] == 4'h0) → S0 (Main Memory)
else if (addr[31:28] == 4'h1) → S1 (DMA Config)
else if (addr[31:28] == 4'h2) → S2 (MMIO)
else                          → 默认错误响应 DECERR
```

### 8.3 Outstanding 支持

- M0（L2 Cache）：支持1个Outstanding读（单MSHR）
- M1（DMA）：支持多个Outstanding写（突发）
- 每个从机均有独立 BID/RID 路由

### 8.4 AXI4-Lite（MMIO/DMA Config）

AXI4-Lite 简化版，无 burst，无 ID，无 WLAST，32位数据：
```
状态机：IDLE → AW_W_LATCH → WRITE_EXEC → B_RESP → IDLE
        IDLE → AR_LATCH   → READ_EXEC  → R_RESP  → IDLE
```

## 第9章 DMA Engine

### 9.1 寄存器接口

| 偏移 | 名称 | 说明 |
|------|------|------|
| 0x00 | CTRL | bit0=start, bit1=abort |
| 0x04 | STATUS | bit0=busy, bit1=done, bit2=error |
| 0x08 | SRC_ADDR | 源地址（字对齐） |
| 0x0C | DST_ADDR | 目的地址（字对齐） |
| 0x10 | LENGTH | 传输字数 |
| 0x14 | INT_EN | bit0=done_ie, bit1=err_ie |

### 9.2 DMA 状态机

```
IDLE ──[start=1]──▶ CALC_SLICE ──▶ RD_REQ ──▶ RD_BURST
                         ▲              │
                         │              ▼
                    [more slices]  WR_REQ ──▶ WR_BURST
                         │              │
                         └──────────────┘
                                        │[all done]
                                        ▼
                                    DONE ──▶ IDLE（产生中断）
```

### 9.3 4KB 边界切片伪代码

```python
def dma_transfer(src, dst, length):
    remaining = length
    while remaining > 0:
        # 计算到下一个4KB边界的距离
        src_boundary = 0x1000 - (src & 0xFFF)
        dst_boundary = 0x1000 - (dst & 0xFFF)
        # 本次传输量 = min(剩余量, src边界, dst边界)
        chunk = min(remaining, src_boundary // 4, dst_boundary // 4)
        axi_burst_read(src, chunk)   # 发起AXI突发读
        axi_burst_write(dst, chunk)  # 发起AXI突发写
        src += chunk * 4
        dst += chunk * 4
        remaining -= chunk
    assert_irq_done()
```

### 9.4 突发参数

- ARBURST/AWBURST = INCR（地址递增）
- ARLEN/AWLEN = chunk_size - 1（最大255 beats）
- ARSIZE/AWSIZE = 3'b010（4字节/beat）

### 9.5 完成中断

DMA 完成后置 STATUS[done]=1，若 INT_EN[done_ie]=1 则拉高 irq 信号送 PLIC。

## 第11章 中断与同步策略

### 11.1 异步复位同步释放

```verilog
// 两级同步器（推荐写法）
reg rst_meta, rst_sync;
always @(posedge clk or negedge rst_n_async) begin
    if (!rst_n_async) begin
        rst_meta <= 1'b0;
        rst_sync <= 1'b0;
    end else begin
        rst_meta <= 1'b1;
        rst_sync <= rst_meta;
    end
end
// rst_sync 为各模块复位信号（同步释放，异步断言）
```

**原因**：异步断言保证复位及时（不等时钟），同步释放避免触发器在不确定时刻退出复位（亚稳态）。

### 11.2 CDC（跨时钟域）

本设计为单时钟域（clk），无跨时钟域信号。若添加外部接口（UART等），需在接口处添加双触发器同步器。

### 11.3 中断流程

```
DMA完成 → irq_done ──▶ plic_lite ──▶ cpu_irq ──▶ CPU异常入口
                    ↑
            INT_EN[done_ie]=1 时使能
```

## 第12章 完整工作流示例

**场景：CPU 写 MMIO 启动 4KB DMA 传输**

```
Cycle  事件
─────────────────────────────────────────────────────────────
  0    CPU 发出 SW 指令，目标地址 0x1000_0008（SRC_ADDR）
  1    DCache 判断 addr[31:28]=1，NC bypass，直接进入NC_BYPASS状态
  2    DCache 发 AXI-Lite AW + W 到 Interconnect
  3    Interconnect 地址解码 → S1 (DMA Config)
  4    dma_regfile 锁存 AW，接受 W，写入 SRC_ADDR 寄存器
  5    B_RESP 返回 CPU，CPU 继续写 DST_ADDR (0x0C)
  ...
  10   CPU 写 CTRL[0]=1（start），DMA 开始
  11   DMA 进入 CALC_SLICE：计算首个切片大小（不跨4KB边界）
  12   DMA 发出 AXI4 AR（ARLEN=切片长度-1）到 Interconnect → S0 (Main Mem)
  13   axi_ram 接受请求，返回突发读数据（RVALID连拍）
  14   DMA 缓存数据，同时发出 AW+W 突发写到 S0 DST地址
  15   写响应 BRESP=OKAY 返回
  16   DMA 检查剩余长度，若>0 返回 CALC_SLICE（下一切片）
  ...
  N    所有切片完成，STATUS[done]=1，irq 拉高
  N+1  PLIC 向 CPU 发中断，CPU 跳转 ISR
  N+2  CPU ISR 读 STATUS 确认，清除中断（写1清0）
```

## 第16章 Q&A 全收录

### Q1：为什么选择写回（Write-Back）而非写穿（Write-Through）？

**A：** 写回策略在写命中时只更新 Cache，不立即写入下级存储，显著降低总线带宽压力。在写密集场景（如矩阵乘法），写穿每次命中都会产生一次总线写事务，容易造成 L2 及主存总线饱和。写回策略将多次写合并，仅在缓存行替换时写回，平均写总线流量降低至原来的 1/（平均命中次数）。本SoC的DCache和L2均采用写回，L1 ICache只读无需考虑写策略。

### Q2：L2 仲裁为何使用 Round-Robin 而非固定优先级？

**A：** 固定优先级在高负载下会导致低优先级方（如L1I）长期饿死（Starvation），影响取指效率从而降低IPC。Round-Robin 保证两者交替获得服务，最大等待时间可界（bounded latency），对实时性要求有保障。在典型场景下L1D与L1I的缺失率相近，轮询能达到近似最优公平性。

### Q3：互联为什么选择 AXI4 而非 AHB 或 APB？

**A：** 
- AHB 为单通道协议，读/写共享总线，吞吐量受限，不支持 Outstanding 事务
- APB 为低速外设总线，无突发，适合MMIO但不适合缓存/DMA高带宽路径
- AXI4 五通道分离（AR/R/AW/W/B），读写并行，支持 Outstanding、乱序响应（通过ID）、突发长度最大256 beats，满足 L2 Cache 缺失填充（16word burst）和 DMA 高带宽需求

### Q4：MMIO 和 DMA Config 为何使用 AXI-Lite？

**A：** MMIO 和 DMA 配置寄存器均为低频访问（CPU显式配置），不需要突发传输，数据宽度固定为32位。AXI-Lite 去除了 ID、WLAST、burst 等字段，逻辑面积更小，从机实现更简单，且完全符合 AXI4 协议超集（AXI-Lite master 可接入 AXI4 互联）。

### Q5：DMA 为什么要做 4KB 边界切片？

**A：** AXI4 协议明确禁止突发事务跨越 4KB 地址边界（4KB boundary crossing），违反此规则会导致地址路由错误（burst 中途被路由到不同从机）。DMA 在计算每次突发前，须检查源地址和目的地址到各自下一个 4KB 边界的距离，取两者与剩余长度的最小值作为本次突发大小，确保合规。

### Q6：性能计数器如何实现？会影响关键路径吗？

**A：** 性能计数器为简单递增寄存器，挂接于 Cache 的 hit/miss 脉冲信号。计数器宽度32位，计数逻辑为 `cnt <= cnt + 1`，是一个加法器链，延迟约 1~2 个逻辑级，不在关键路径（Cache访问/AXI握手）上。计数器挂接于输出端，不影响 Cache 命中判断的时序。

### Q7：l2_arbiter 为什么设计为独立模块？

**A：** 参见第7章。核心原因是关注点分离：仲裁策略与缓存数据通路正交，独立模块使得两者可以分别优化和验证。在设计迭代中，若需要将 Round-Robin 替换为加权仲裁（Weighted Arbitration），只需修改 `l2_arbiter.v`，无需改动 `l2_cache.v`，降低回归风险。

### Q8：异步复位同步释放的具体好处是什么？

**A：** 
- **异步断言**：复位信号到来时无需等待时钟，所有触发器立即进入复位状态，适用于上电初始化及掉电场景
- **同步释放**：复位释放时，经过两级触发器同步后统一退出复位，避免不同触发器在不同时钟沿退出复位（时序发散），防止亚稳态传播
- **实现代价**：仅需两个额外触发器，面积代价极小

### Q9：本设计的 Cache 一致性策略是什么？

**A：** 本SoC为单核设计，不存在多核一致性问题。DMA 与 CPU 共享主存时存在潜在一致性问题：
- **DMA写→CPU读**：DMA 直接写主存（不经过 Cache），CPU L1/L2 中可能有旧数据。解决方案：CPU 在 DMA 完成中断 ISR 中对目标地址范围执行 Cache flush/invalidate
- **CPU写→DMA读**：CPU 写入 DCache（写回），DMA 可能读到主存中的旧数据。解决方案：CPU 在启动 DMA 前 flush DCache 中的脏行
- 本设计通过软件约定（驱动层保证flush）实现一致性，不使用硬件 snooping/MOESI，以降低硬件复杂度

### Q10：为什么 L1I 选择 4路而 L1D 只有 2路？

**A：** 指令访问具有高度局部性（循环、顺序执行），4路组相联能有效减少冲突缺失（Conflict Miss），提升命中率。DCache 访问模式更随机（数据地址分散），增加路数的收益边际递减，而2路已能覆盖典型的时间局部性。此外，2路 LRU 替换可以精确实现（1 bit per set），而4路精确 LRU 需要 log2(4!)=4.58 bit，通常退化为 PLRU，DCache 采用2路 LRU 更精确。
